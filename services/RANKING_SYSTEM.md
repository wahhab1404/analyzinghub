# Ranking & Reputation System Documentation

## Overview

A transparent, auditable points and badge system for AnalyzingHub that rewards:
- **Analysts**: Accurate predictions and validated outcomes
- **Traders**: Quality engagement and insightful participation

All scoring is system-calculated with anti-gaming protections to ensure fairness.

## Architecture

### Core Components

1. **Ledger System** - Immutable log of all point events (source of truth)
2. **Balance Cache** - Fast access to current point totals
3. **Badge Engine** - Automated badge awarding based on performance
4. **Leaderboards** - Cached rankings (weekly, monthly, all-time)
5. **Anti-Gaming** - Multiple layers of fraud prevention

### Database Schema

#### `user_points_ledger`
Immutable event log for all point-generating actions
- Ensures idempotency (same event can't score twice)
- Full audit trail with metadata
- Unique constraints prevent duplicate scoring

#### `user_points_balance`
Cached point totals for fast queries
- All-time, weekly, and monthly points
- Separate for analyst and trader roles
- Updated incrementally from ledger

#### `user_stats`
Cached statistics for badge calculations
- Win rates, closed analyses, consecutive stops
- Rating accuracy, engagement metrics
- Recalculated nightly

#### `user_badges`
Badge awards with revocation support
- Awarded automatically based on criteria
- Can be revoked if criteria no longer met
- Metadata tracks achievement details

#### `leaderboard_cache`
Pre-computed leaderboard data
- Separate for analysts/traders and time scopes
- Regenerated hourly or on-demand
- Includes quality scores and metadata

## Scoring Rules

### Analyst Points

| Event | Points | Requirements |
|-------|--------|-------------|
| Analysis Created | +5 | Valid analysis with targets and stop loss |
| Target Hit | +10 | Per target, system-validated |
| Stop Loss Hit | -10 | System-validated |

**Validation Requirements:**
- Analysis must have: symbol, direction, stop loss, at least 1 target
- Validation events generated by platform (not user input)
- Analysis immutable after publication (no editing targets/stop)

**Daily Caps:**
- Maximum 10 analyses per day (50 points max from creation)
- No cap on target/stop validations (organic market events)

### Trader Points

| Event | Points | Requirements |
|-------|--------|-------------|
| Like | +1 | Unique per analysis per user |
| Bookmark | +2 | Unique per analysis per user |
| Repost | +3 | Unique per analysis per user |
| Comment | +3 | Quality rules (see below) |
| Rating | +5 | Analysis must be closed, not own analysis |

**Comment Quality Rules:**
- Minimum 25 characters
- Not repeated (checked against recent hour)
- Not only emojis (max 50% emoji ratio)
- Max 3 links
- At least 5 words

**Daily Caps:**
- Maximum 100 points per day from trader activities
- Prevents spam and gaming

**Account Requirements:**
- Account must be 7+ days old
- Email must be verified
- Applies to all trader point activities

## Anti-Gaming Protections

### 1. Account Age Restrictions
- New accounts (<7 days) cannot earn trader points
- Email verification required
- Prevents throwaway account abuse

### 2. Daily Caps
- Trader: 100 points/day maximum
- Analyst: 10 analyses/day creation limit
- Caps reset at midnight UTC

### 3. Uniqueness Constraints
- Database unique indexes on (user_id, entity_id, event_type)
- Prevents:
  - Liking same analysis twice
  - Bookmarking same analysis twice
  - Reposting same analysis twice
  - Rating same analysis twice

### 4. Quality Scoring
- Comment content analyzed for quality
- Emoji/link ratios checked
- Duplicate detection in recent hour
- Word count minimums

### 5. Rate Limiting
- High-volume activity flagged (>20 actions/minute)
- Suspicious users logged for review
- Temporarily excluded from leaderboards

### 6. Immutable Ledger
- All points derived from logged events
- No manual point adjustments possible
- Full audit trail for investigation

## API Endpoints

### GET /api/leaderboards
Query Parameters:
- `type`: 'analyst' | 'trader'
- `scope`: 'weekly' | 'monthly' | 'all_time'

Returns top 100 users with points, stats, and badges.

**Caching:** Results cached for 1 hour. Force refresh by calling recalculate job.

### GET /api/rankings/[id]
Returns complete ranking data for a user:
- Analyst stats (points, rank, win rate, badges)
- Trader stats (points, rank, engagement, badges)
- All-time and time-scoped points

### POST /api/ratings/submit
Body: `{ analysisId, rating: 1-5 }`

Creates rating and awards 5 points if:
- Analysis is closed (SUCCESS or FAILED)
- Not rating own analysis
- Haven't rated this analysis before

### POST /api/scoring/award
Internal endpoint for awarding points.

Headers: `X-API-Key: [INTERNAL_API_KEY]`
Body: `{ eventType, userId, entityId, metadata }`

Used by system to award points for events. **Not for client use.**

### POST /api/scoring/recalculate
Triggers nightly recalculation job.

Headers: `X-API-Key: [INTERNAL_API_KEY]`

**Runs:**
- Update point balances from ledger
- Calculate user statistics
- Evaluate and award/revoke badges
- Regenerate leaderboard cache
- Clean old daily cap records

## Scheduled Jobs

### Nightly Recalculation (Recommended: 2 AM UTC)

```bash
# Using cron
0 2 * * * curl -X POST https://your-domain.com/api/scoring/recalculate \
  -H "X-API-Key: YOUR_INTERNAL_KEY"
```

**What it does:**
1. Recalculates all point balances (all-time, weekly, monthly)
2. Updates user statistics (win rates, engagement metrics)
3. Evaluates badge criteria for all users
4. Awards new badges and revokes outdated ones
5. Regenerates all leaderboard caches
6. Cleans up old daily cap records (>7 days)

**Duration:** ~2-5 minutes for 1000 users

### Alternative: Supabase Edge Function with Cron

```typescript
// supabase/functions/nightly-scoring/index.ts
Deno.serve(async () => {
  const response = await fetch(`${Deno.env.get('APP_URL')}/api/scoring/recalculate`, {
    method: 'POST',
    headers: {
      'X-API-Key': Deno.env.get('INTERNAL_API_KEY') || '',
    },
  })

  return new Response(await response.text())
})
```

Enable in Supabase Dashboard → Edge Functions → Cron Jobs

## Performance Considerations

### Database Indexes
All critical foreign keys and query columns are indexed:
- `user_points_ledger(user_id, event_type, created_at)`
- `user_points_balance(user_id)`
- `user_stats(win_rate, rating_accuracy)`
- `leaderboard_cache(scope, type, rank)`

### Query Optimization
- Leaderboards served from cache (no expensive calculations)
- User stats pre-calculated nightly
- Balance updates incremental (not full recalculation)

### Caching Strategy
- Leaderboards cached for 1 hour
- User stats cached until nightly job
- Balances updated immediately but queried from cache

## Monitoring

### Key Metrics to Track

1. **Point Distribution**
   - Average points per user
   - Points by event type
   - Daily point volume

2. **Badge Achievement Rates**
   - % of users with badges
   - Time to first badge
   - Badge distribution by tier

3. **Suspicious Activity**
   - Flagged users per day
   - Most common suspicious patterns
   - False positive rate

4. **System Health**
   - Ledger insert failures
   - Daily cap violations
   - Recalculation job duration

### SQL Queries for Monitoring

```sql
-- Point distribution
SELECT event_type, COUNT(*), SUM(points_delta), AVG(points_delta)
FROM user_points_ledger
WHERE created_at >= NOW() - INTERVAL '7 days'
GROUP BY event_type;

-- Top scorers
SELECT p.full_name, upb.analyst_points_all_time, upb.trader_points_all_time
FROM user_points_balance upb
JOIN profiles p ON upb.user_id = p.id
ORDER BY (analyst_points_all_time + trader_points_all_time) DESC
LIMIT 10;

-- Badge distribution
SELECT badge_key, badge_tier, COUNT(*)
FROM user_badges
WHERE revoked_at IS NULL
GROUP BY badge_key, badge_tier
ORDER BY COUNT(*) DESC;

-- Suspicious activity
SELECT activity_type, COUNT(*)
FROM suspicious_activity_log
WHERE flagged_at >= NOW() - INTERVAL '24 hours'
AND reviewed_at IS NULL
GROUP BY activity_type;

-- Daily cap violations
SELECT date, COUNT(DISTINCT user_id), SUM(trader_points_today)
FROM daily_points_cap
WHERE date >= CURRENT_DATE - 7
GROUP BY date
ORDER BY date DESC;
```

## Troubleshooting

### Issue: Points not being awarded

1. Check ledger insert errors in logs
2. Verify unique constraints not violated
3. Check daily caps for user
4. Verify account age and email verification
5. Test idempotency - may already be awarded

### Issue: Badges not updating

1. Run manual recalculation: `POST /api/scoring/recalculate`
2. Check user stats calculation: `SELECT * FROM user_stats WHERE user_id = ?`
3. Verify win rate calculation logic
4. Check closed analyses count

### Issue: Leaderboard not updating

1. Check cache generation time
2. Force refresh with recalculation job
3. Verify balance updates are working
4. Check RLS policies for leaderboard_cache

### Issue: Daily caps not resetting

1. Verify date comparison in queries
2. Check cleanup job is running
3. Manually delete old records if needed
